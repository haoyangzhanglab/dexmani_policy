# 08-审查约定

与上游 Diffusion Policy / DP3 / ManiFlow_Policy 对齐的已知设计模式，以及经两轮全面审查确认的架构决策。审查时应避免误报。

---

## 一、数据与 Normalizer

### 1.1 Normalizer 在全量数据上拟合

**文件**: `datasets/base_dataset.py:get_normalizer()`、`datasets/pc_dataset.py:get_normalizer()`

`BaseDataset.get_normalizer()` 使用 `self.replay_buffer` 的全部数据（含验证集 episodes）拟合 normalizer 参数，而非仅在训练集上拟合。只拟合 `joint_state` 和 `action` 两个 keys，`point_cloud` / `rgb` 等原始传感器数据不经过 `LinearNormalizer`。

**这不是数据泄露**：
- Imitation learning 中所有数据来自同一批专家演示，train/val 是随机切分而非异分布
- `mode='limits'` 只捕获 min/max，验证集几乎不会改变归一化边界
- 评测指标是 simulation success rate，不依赖 val loss 的精确性
- Diffusion Policy → DP3 → ManiFlow_Policy 全系列均采用此做法

**上游客栈证据**:
- `real-stanford/diffusion_policy` `pusht_dataset.py:get_normalizer()`: 全量 `self.replay_buffer` 传入 `normalizer.fit()`
- `geyan21/ManiFlow_Policy` `maniflow_pointcloud_policy.py`: 同模式

### 1.2 Point Cloud / RGB 不进入 LinearNormalizer

`PCDataset.get_normalizer()` 和 `RGBDataset.get_normalizer()` 只对 `joint_state` 和 `action` 拟合。`LinearNormalizer.normalize()` 中未拟合的 key（`point_cloud`、`rgb`）原样透传。点云坐标中心化和图像 ImageNet 标准化由各自 encoder 内部处理。

灵巧操作中，点云坐标 (X,Y,Z) 的全局 min/max 归一化会破坏物体几何形状的相对关系（如抓取点的局部曲率）。保留原始坐标空间由 encoder 的 local ops（如 PointNet 的 T-Net / 球查询）处理是正确的。

### 1.3 ReplayBuffer 内存加载时统一转 float32

**文件**: `datasets/common/replay_buffer.py:copy_from_store()`（numpy 分支）

对所有浮点类型数组统一执行 `astype(np.float32)`。`point_cloud` / `rgb` 等不经过 `LinearNormalizer` 的模态，若不转 float32，可能以 float64 进入 encoder，造成显存翻倍。

zarr backend（磁盘模式）不做此转换，因 zarr 自身 dtype 管理机制更完善。标准训练路径（`copy_from_path` → `copy_from_store` with `store=None`）走 numpy backend，不存在 float64 残留问题。

### 1.4 LinearNormalizer 零方差维度处理

**文件**: `common/normalizer.py:fit_params()`

当某维度所有值相同时（如固定关节角），`input_range=0` 触发 `ignore_dim`（`range < range_eps=2e-2`）：
- `input_range[ignore_dim] = 2.0`（防除零）
- `scale = 1.0`, `offset = -input_mean`（零中心化，不缩放）

归一化输出 ≈ 0，逆归一化时 scale=1.0 不产生 NaN/Inf。19-DOF 灵巧手常有手指末端关节运动范围极小的情况，此机制确保训练稳定。

### 1.5 MultiTask 共享 Normalizer 跨任务拟合

**文件**: `datasets/multi_task_dataset.py:compute_shared_normalizer()`

`normalizer_mode='shared'` 时，所有子任务的 `joint_state` 和 `action` 沿第 0 维 concat 后统一拟合。min/max 取自所有任务全局极值，`[-1, 1]` 覆盖完整值域。

`normalizer_mode='per_task'` 在 `train.py:33-38` 被 `NotImplementedError` 拦截，当前不可用。

### 1.6 SequenceSampler 短 episode 预过滤

**文件**: `datasets/common/sampler.py:SequenceSampler.__init__()`

在调用 `create_indices()` 前预检查每条 episode 长度，短于 `min_required_length = sequence_length - pad_before - pad_after`（默认 8 帧）的 episode 被 warn + skip。若所有 episode 均被过滤则抛 `ValueError`。

上游 Diffusion Policy 对短 episode 的处理是 `continue`（跳过），本项目改为 `warn + mask out` 语义等价。灵巧操作演示中可能出现异常短的无效轨迹片段（如环境 reset），此容错确保训练可启动。

## 二、训练机制

### 2.1 NaN 检测三层防护

**文件**: `training/trainer.py`、`training/ddp_trainer.py`

| 层级 | 位置 | 机制 |
|---|---|---|
| ① Loss NaN | `trainer.py:train_one_step()` / `ddp_trainer.py:train()` | `torch.isfinite(loss)` 检查，非有限时 `zero_grad` 清空梯度后抛异常 |
| ② Gradient NaN | `trainer.py:apply_gradient_step()` | `clip_grad_norm_` 返回 total_norm 的 NaN 检查 |
| ③ DDP 同步 | `ddp_trainer.py:train()` | 每步 `dist.all_reduce(nan_flag, MAX)` in `backward()` 前，防止单 rank NaN 导致集群死锁 |

DDP 场景下 NaN 检测必须在 `backward()` 之前完成 collective 同步：若 rank A 检测到 NaN 直接抛异常，rank B/C/D 的 `backward()` 会 enqueue gradient all-reduce，因 rank A 不参与而永久阻塞。当前实现在 `compute_loss` 后立即 `all_reduce(nan_flag)`，所有 rank 一致退出。

### 2.2 Epoch 边界梯度累积 Flush

**文件**: `training/trainer.py:flush_gradient_accumulation()`；`ddp_trainer.py:epoch` 循环末尾

当 epoch 结束时仍有未完成的梯度累积（`accum_step % grad_accum_steps != 0`），`flush` 将当前累积的梯度乘以 `grad_accum_steps / (accum_step % grad_accum_steps)` 后再执行 `optimizer.step()`。

缩放因子确保部分累积与完整累积的梯度尺度一致。与 `total_steps = ceil(batches_per_epoch / grad_accum_steps)` 的 ceil 逻辑一致，无 off-by-one。

### 2.3 FlowMatch 验证阶段使用 Student 模型（非 EMA）

**文件**: `training/trainer.py:finish_epoch()` 的 `validate` 分支

当 `use_ema_teacher_for_consistency=True` 时，验证阶段使用训练模型（student）计算 loss，EMA 模型仅作为 consistency teacher。与 DP/DP3 不同（DP/DP3 验证时优先使用 EMA），FlowMatch 必须保持 student ≠ teacher 才能正确计算 consistency loss（否则退化为平凡自洽映射）。

### 2.4 Checkpoint 文件累积防护

**文件**: `training/trainer.py:finish_epoch()`

当 `save_latest=True` 而 `save_checkpoint=False`（仅验证不评测的 epoch），不创建新 .pt 文件，改为查找已有 checkpoint 并更新 latest symlink。避免未评测 epoch 产生无 tracker 管理的孤立文件。

### 2.5 NaN Debug 检查点自动保存

**文件**: `training/trainer.py:_save_nan_debug()` / `train_one_step()`

`train_one_step()` 在 loss 非有限值时自动保存 checkpoint（含 model/ema/optimizer/scheduler state），文件名格式 `nan_debug_epoch=XXXX_step=XXXXXXXX_YYYYMMDD_HHMMSS.pt`。保存完即 raise，不吞异常。这是生产级数值不稳定性排查机制，非遗留调试代码。

## 三、模型架构

### 3.1 FlowMatch sample_ode relative 模式下 target_t=dt

**文件**: `agents/action_decoders/flowmatch.py:sample_ode()`

`sample_ode` 在 `target_t_sample_mode='relative'` 时，每个 ODE 步的 `target_t` 设为 `dt`（>0）。若训练时未启用 consistency training（`use_ema_teacher_for_consistency=False`），模型仅在 `target_t=0` 下训练，推理时的 `target_t>0` 属于条件分布偏移。

**这不是 bug，是 upstream 原始设计**：
- 上游 `geyan21/ManiFlow_Policy` 的 `sample_ode` 行为完全一致
- 默认配置 `maniflow.yaml` 中 `use_ema_teacher_for_consistency: true` 通过 consistency training 提供 `target_t>0` 泛化
- `train.py:validate_config()` 在未启用 consistency 时发出 UserWarning

**上游客栈证据**:
- `geyan21/ManiFlow_Policy` `maniflow_pointcloud_policy.py:sample_ode()`: relative 模式下 `target_t=dt`

### 3.2 FlowMatch 速度场方向与 ODE 积分

**文件**: `agents/action_decoders/flowmatch.py:get_flow_velocity()`、`sample_ode()`

- 训练目标速度：`vt_target = x1 - x0`（rectified flow 直线路径的解析导数 `dx/dt`）
- ODE 积分：Euler 步 `x_{t+dt} = x_t + v_pred * dt`，从 t=0（噪声）积分到 t=1（数据）
- 完美模型下 `v_pred = x1 - x0`，Euler 沿直线前进，方向与步长正确

### 3.3 FlowMatch Consistency Training 推导

**文件**: `agents/action_decoders/flowmatch.py:get_consistency_velocity()`

Consistency target 由 Teacher EMA 模型估计：
1. Teacher 在 `xt_next` 处以 `target_t=1-t_next` 预测速度 → 外推终态 `pred_x1_ct`
2. Target velocity = `(pred_x1_ct - xt_ct) / (1 - t_ct)`
3. 学生以 `target_t=dt` 预测速度，MSE 匹配 target

当 t_next=1.0 时（约 45% consistency 样本），`xt_next = x1`（ground truth），`pred_x1_ct = x1`（无需 Teacher），target = `(x1 - xt_ct)/(1-t_ct) = x1 - x0`（精确 ground truth）。这为 consistency 训练提供强监督信号。

Teacher 梯度隔离（`torch.no_grad()`）、eval 模式（`ema_model.eval()`）、EMA 权重更新均正确。

### 3.4 FlowMatch compute_loss 批次拆分与边界处理

**文件**: `agents/action_decoders/flowmatch.py:compute_loss()`

| 条件 | 行为 | 原因 |
|---|---|---|
| `ema_model is None` | 全量样本计算 flow loss，consistency loss=0 | EMA 尚未就绪（训练早期或未启用 consistency） |
| `B < 2` | Warning 后全量 flow loss | batch 不足以拆分 flow/consistency（常见于 grad_accum_steps > batch_size 时最后一个不完整 batch） |
| `consistency_batchsize == 0` | flow loss only | 极端情况，无法为 consistency 分配样本 |

batch 拆分：`flow_batchsize = max(1, min(B-1, int(B * flow_batch_ratio)))`，`consistency_batchsize = B - flow_batchsize`。

### 3.5 AdaLNZero 初始化保护

**文件**: `agents/action_decoders/backbone/ditx.py:initialize_weights()`

`DiTXFlowMatch` 和 `DiTXDiffusion` 的 `initialize_weights()` 中，`self.apply(init_fn)` 会递归对所有 `nn.Linear` 执行 xavier_uniform，覆盖 `self.context_norm`（`AdaLNZero`）的特殊初始化（`weight=0, bias=split`）。

修复：在 `self.apply(init_fn)` 后显式调用 `self.context_norm.initialize_weights()` 恢复 AdaLNZero 初始化。AdaLNZero 的核心特性是初始状态下作为 identity（`gamma=1, beta=0`），保证训练早期条件注入不破坏特征分布。

### 3.6 obs_encoder 参数组使用 get_optim_group_with_no_decay

**文件**: `agents/core/base.py:get_optim_param_groups()`

`BaseAgent.get_optim_param_groups()` 复用 `agents/common/optim_util.py` 的 `get_optim_group_with_no_decay(self.obs_encoder, weight_decay=obs_wd)`，按模块类型分拆 decay/no-decay 组（whitelist: Linear/Conv/MultiheadAttention → decay；blacklist: LayerNorm/BatchNorm/GroupNorm/InstanceNorm/Embedding + bias → no_decay）。含 100% 参数覆盖率断言。

与 action_decoder backbone 的 `get_optim_groups()` 使用同一基础函数，确保策略一致。

### 3.7 CLIP Text Encoder 显式冻结

**文件**: `agents/core/multi_task.py:__init__()`

`MultiTaskAgent` 构造函数中 `self.text_encoder.requires_grad_(False)` 显式冻结 CLIP text encoder。text_emb 仅通过 `text_proj`（Linear）可训练。这比"不加入 optimizer"的隐式冻结更安全：DDP wrapper / gradient checkpointing 不会产生不必要的计算图。

### 3.8 MoE 全专家计算与 DDP 兼容性

**文件**: `agents/obs_encoder/plugins/moe.py:aggregate_experts()`

`aggregate_experts()` 通过 `torch.stack([expert(z) for expert in self.experts])` 计算所有 expert 的 forward，再用 `torch.gather + weighted sum` 选取 top-K 结果。非选中 expert 获零梯度（非 None 梯度），满足 DDP `find_unused_parameters=False` 的要求。

这是用计算效率（与 conditional MoE 相比为 `num_experts/top_k` 倍 FLOPs）换取 DDP 兼容性的有意设计。

## 四、DDP 与多卡训练

### 4.1 DDP find_unused_parameters=False

**文件**: `training/ddp_trainer.py:__init__()`

经核实，当前所有模型变体（DP/DP3/ManiFlow/MoE/MultiTask）均与 `find_unused_parameters=False` 兼容：
- MoE 全专家计算确保所有参数获得梯度（见 3.8）
- CLIP 冻结参数 `requires_grad=False` 不在 DDP 管理范围
- FlowMatch consistency teacher 在 `no_grad()` 下调用，不触发 DDP hook

### 4.2 DDP 梯度累积与 All-Reduce 语义等价

**文件**: `training/ddp_trainer.py:train()`

每个 `backward()` 触发 DDP gradient all-reduce（`SUM / world_size`）。loss 已缩放 `1/grad_accum_steps`，累积 K 次后总梯度 = `(1/(K*world_size)) * Σ_i Σ_rank grad_i`，与单卡 `batch_size * grad_accum_steps` 的语义一致。

### 4.3 Normalizer DDP 同步

**文件**: `training/ddp_trainer.py:synchronize_states()`

`state_dict()` 返回 detached tensor（与模型参数共享存储），`dist.broadcast` 原地修改。rank 0 只 broadcast 不加载，rank>0 broadcast 后 `load_state_dict`。

`sorted(norm_state)` 确保所有 rank 上 key 顺序一致；所有 normalizer 状态均为 `nn.Parameter`（tensor），无 string 等非 tensor 状态。

### 4.4 变 world_size Resume 的 LR Schedule 失真

**文件**: `train_ddp.py:build_optimizer_and_scheduler()`、`ddp_trainer.py:train()`

Resume 时 `total_steps` 基于当前 world_size 的 `batches_per_epoch` 重新计算，但 `scheduler.load_state_dict` 恢复了旧的 `_step_count`。当新旧 world_size 不同时，LR 曲线被压缩或拉伸（如 world_size=4→2 时 total_steps 翻倍，LR 曲线被拉平）。

模型权重、EMA、optimizer moments 不受 world_size 影响，仅 LR schedule 受此问题影响。此为低概率场景（用户很少改变 world_size），但如发生需注意。

### 4.5 Fix_state_dict 的 DDP 语义

**文件**: `training/common/checkpoint_io.py:fix_state_dict()`

Checkpoint 始终以 `raw_model`（unwrapped）保存，不含 `module.` 前缀。Resume 时 `is_current_ddp=False` → `fix_state_dict` 不做任何转换（no-op）。加载到 raw_model 再 wrap DDP。

## 五、增强与预处理

### 5.1 PCSpatialAug 对 XYZ + 法向通道施加同步旋转

**文件**: `datasets/augmentation/pc_spatial.py:_augment()`

`PCSpatialAug` 对 XYZ 通道施加 Z 轴旋转 + XY 平移 + uniform 缩放。当 `pc_dim >= 6` 时，通道 3:6（法向）自动同步旋转（不含平移/缩放）。颜色通道不受空间变换影响，由 `PCColorJitter` 独立处理。

当前所有配置使用 `pc_dim=3`（仅 XYZ），法向旋转逻辑不触发。

### 5.2 PCColorJitter RGB 通道位置

**文件**: `datasets/augmentation/pc_color.py:_augment()`

`PCColorJitter` 的 docstring 声明 RGB 在 channels `[-3:]`，实现已修正为此语义（原硬编码 `[..., 3:6]` 在 pc_dim=9 (XYZ+Normal+RGB) 时会错误修改法向）。修正后对任意 `pc_dim >= 6` 格式均正确处理最后 3 通道为 RGB。

### 5.3 Modality Dropout 仅对已拟合模态生效

**文件**: `agents/core/base.py:preprocess()`

`modality_dropout` 仅对 `k in normalizer.params_dict` 的模态生效（即 `joint_state`）。`action`、`point_cloud`、`rgb` 的 dropout 不在此路径处理。`train.py:validate_config()` 对 `action` 的 dropout 配置有显式告警。

## 六、MultiTask

### 6.1 MultiTaskDataset epoch 跨 worker 同步

**文件**: `datasets/multi_task_dataset.py`

`MultiTaskDataset` 使用 `multiprocessing.Value('i', 0)` 存储当前 epoch。`set_epoch()` 同时更新 `self._epoch` 和 `self._epoch_val.value`，`__getitem__` 在 worker 中从 `self._epoch_val.value` 读取。

解决 `persistent_workers=True` 下 worker 进程无法感知主进程 epoch 变更的问题（worker fork 后继承 shared memory 引用，主进程写入对 worker 可见）。

### 6.2 Deterministic 索引生成使用 epoch=0

**文件**: `datasets/multi_task_dataset.py:generate_fixed_indices()`、`compute_target_counts()`

`generate_fixed_indices()` 调用 `compute_target_counts()` 时 `self._epoch=0`（初始化值）。diff 修正的分配规律与 epoch=0 的非 deterministic 模式相同。索引的 MD5 hash 保证确定性，实验可复现。

### 6.3 per_task success_rate 单位统一

**文件**: `training/trainer.py:evaluate()`

`MultiTaskSimRunner.run()` 返回的 `per_task/{name}/success_rate` 为小数 (0-1)，`evaluate()` 在存储日志时统一乘以 100 转换为百分比，与 `eval/success_rate` 保持量纲一致。

## 九、灵巧操作特化约定

### 9.1 19-DOF 动作空间的归一化语义

动作空间为 XArm7 (7-DOF) + XHand (12-DOF) 的绝对关节位置，共 19 维。`LinearNormalizer` 的 `last_n_dims=1` 对每个关节独立归一化，不耦合不同关节的量纲。

XArm7 关节范围（rad）与 XHand 手指关节范围（线性/角度混合）差异可达数量级。Per-dim 归一化确保低范围维度（如手指末端关节）不会被高范围维度（如手臂基座关节）的 loss 主导。

### 9.2 点云通道布局

| pc_dim | 布局 | 配置 |
|---|---|---|
| 3 | XYZ | 所有当前配置 |
| 6 | XYZ + Normal (nx, ny, nz) | 可扩展 |
| 9 | XYZ + Normal + RGB | 可扩展 |

`PCSpatialAug` 和 `PCColorJitter` 已兼容 pc_dim>=6，但当前训练路径不触发。

### 9.3 Control Action 时序切片

`pred[:, n_obs_steps-1 : n_obs_steps-1+n_action_steps]` = `pred[:, 1:9]`（horizon=16, n_obs_steps=2, n_action_steps=8）。

这是 Diffusion Policy 的标准 convention：前 2 步为观测上下文，模型预测未来 16 步轨迹，实际执行 8 步后重新推理（receding horizon control）。灵巧操作中 8 步动作缓冲足够覆盖抓取→移动→释放的单阶段动作。

### 9.4 一致性训练对灵巧操作的适应性

FlowMatch consistency training 对灵巧操作有特殊价值：
- 灵巧手的接触动力学高度非线性，速度场在 t→1（接近真实轨迹）处可能剧烈变化
- Consistency training 的离散采样点 `{0, 0.1, ..., 0.9}` 恰好覆盖推理时的精确位置
- t_next=1.0 时的 ground truth 监督（见 3.3）为接近数据流形的推理步提供精确指引
