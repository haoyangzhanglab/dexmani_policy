## 架构概览

```
train.py (单卡入口)          train_ddp.py (多卡入口)
    ↓                              ↓
Trainer (核心训练循环)      DDPTrainer (DDP 包装器)
    ↓                              ↓
TrainWorkspace              ReadOnlyWorkspace (非主进程)
```

**设计原则**: 单卡和多卡训练完全解耦，`Trainer` 不知道 DDP 的存在。

---

## 核心组件

### 1. Trainer (单卡训练)

**职责**: 纯粹的训练循环逻辑，不包含任何分布式相关代码。

**关键方法**:
- `train_one_step()` - 单步训练（梯度累积 + 优化器更新 + EMA 更新）
- `validate()` - 验证集评估
- `evaluate()` - 仿真环境评估
- `train()` - 完整训练循环

**特点**:
- 直接使用 `self.model`（可能是 DDP 包装的，但 Trainer 不知道）
- 不依赖任何 DDP 工具函数
- 可独立用于单卡训练

---

### 2. DDPTrainer (多卡训练)

**职责**: DDP 包装和分布式协调，所有 DDP 逻辑都在这里。

**关键设计**:
```python
self.raw_model = model  # 保存原始模型引用
ddp_model = DDP(model, ...)  # 包装为 DDP
self.trainer = Trainer(model=ddp_model, ...)  # 传递 DDP 模型给 Trainer
```

**DDP 逻辑处理**:
- **EMA 更新**: Trainer 直接使用 DDP 模型（`named_parameters()` 不带前缀）
- **Checkpoint 保存**: 使用 `self.raw_model.state_dict()`（无 `module.` 前缀）
- **验证/评估**: 传递 `self.raw_model` 或 `ema_model`（都未包装）
- **Normalizer 同步**: 使用 `dist.broadcast` 同步到所有 rank

**关键方法**:
- `synchronize_states()` - 同步 normalizer 状态
- `train()` - DDP 训练循环（包含 barrier 同步）

## 训练流程

### 单卡训练流程

```
1. 初始化组件 (model, optimizer, scheduler, ema, workspace)
2. 加载 checkpoint (如果存在)
3. 训练循环:
   for epoch in epochs:
       for batch in train_loader:
           - 前向传播 + 反向传播
           - 梯度累积
           - 优化器更新 + EMA 更新
       - 验证 / 评估 / 保存 checkpoint
```

### 多卡训练流程

```
1. 每个 rank 初始化组件
2. rank 0 创建 TrainWorkspace，其他 rank 创建 ReadOnlyWorkspace
3. 所有 rank 加载 checkpoint (如果存在)
4. 同步 normalizer 状态 (dist.broadcast)
5. 训练循环:
   for epoch in epochs:
       train_sampler.set_epoch(epoch)  # 确保 shuffle 随机性
       for batch in train_loader:
           - 所有 rank 执行训练步
           - DDP 自动同步梯度
       dist.barrier()  # 同步所有 rank
       if rank == 0:
           - 验证 / 评估 / 保存 checkpoint
       dist.barrier()  # 等待主进程完成
```

## Checkpoint 机制

### TrainCheckpoint 结构

```python
@dataclass
class TrainCheckpoint:
    epoch: int
    global_step: int
    model_state: Dict          # 模型权重（无 module. 前缀）
    ema_model_state: Dict      # EMA 模型权重
    ema_updater_state: Dict    # EMA updater 状态 (decay, optimization_step)
    optimizer_state: Dict      # 优化器状态
    scheduler_state: Dict      # 学习率调度器状态
    monitor: Dict              # 监控指标 (test_mean_score)
```

### 保存策略

- **latest.pt**: 最新 checkpoint（符号链接）
- **topk**: 根据 `test_mean_score` 保存 top-k checkpoint
- **命名格式**: `epoch=0042-step=00012345-score=0.8500.pt`

### State Dict 前缀处理

**问题**: DDP 模型的 state_dict 包含 `module.` 前缀，需要统一处理。

**解决方案**:
- **保存**: 使用 `raw_model.state_dict()`（无前缀）
- **加载**: 使用 `fix_state_dict()` 自动处理前缀
- **兼容性**: 单卡 ↔ 多卡 checkpoint 完全兼容

## EMA 机制

### 设计

```python
class EMAModel:
    def __init__(self, model, power=2/3, max_value=0.9999):
        self.averaged_model = model
        self.decay = 0.0
        self.optimization_step = 0
    
    def step(self, new_model):
        self.decay = self.get_decay(self.optimization_step)
        # ema_param = ema_param * decay + new_param * (1 - decay)
        self.optimization_step += 1
```

### 更新时机

- **单卡**: `Trainer.train_one_step()` 中优化器更新后
- **多卡**: 同样在 `Trainer.train_one_step()` 中，直接使用 DDP 模型

**重要**: DDP 模型的 `named_parameters()` **不带** `module.` 前缀，所以可以直接用于 EMA 更新。

### 使用场景

- **训练**: 使用主模型（可能是 DDP 包装）
- **验证**: 使用 EMA 模型（未包装）
- **评估**: 使用 EMA 模型（未包装）
- **Checkpoint**: 同时保存主模型和 EMA 模型

## 梯度累积

### 实现

```python
loss = raw_loss / gradient_accumulate_every
loss.backward()
self.accum_step += 1

if self.accum_step % gradient_accumulate_every == 0:
    clip_grad_norm_(self.model.parameters(), max_norm=grad_clip_norm)
    optimizer.step()
    scheduler.step()
    optimizer.zero_grad(set_to_none=True)
    ema_updater.step(model)  # 只在优化器更新后更新 EMA
```

### 学习率调度器步数计算

```python
batches_per_epoch = len(train_loader)
optimizer_steps_per_epoch = (batches_per_epoch + gradient_accumulate_every - 1) // gradient_accumulate_every
total_steps = optimizer_steps_per_epoch * num_epochs
```

## Workspace 设计

### TrainWorkspace (主进程)

**职责**: 日志记录、checkpoint 管理、配置保存。

**组件**:
- `MultiLogger` (JsonlLogger + WandbLogger)
- `CheckpointStore` (原子性保存)
- `TopKCheckpointTracker` (top-k 管理)

### ReadOnlyWorkspace (非主进程)

**职责**: 只读 checkpoint 加载，不执行任何写操作。

**特点**:
- `load_checkpoint()` - 可以加载
- `save_*()` - 返回 dummy 值
- `log()` - 空操作

## DDP 关键设计

### 1. 只包装训练模型

```python
# ✓ 正确
model = DDP(model, ...)
ema_model = ema_model  # 不包装

# ✗ 错误
model = DDP(model, ...)
ema_model = DDP(ema_model, ...)  # EMA 不需要梯度同步
```

### 2. EMA 更新直接使用 DDP 模型

```python
# Trainer.train_one_step() 中
if self.use_ema and self.ema_updater is not None:
    self.ema_updater.step(self.model)  # self.model 可能是 DDP 包装的
```

**原理**: DDP 模型的 `named_parameters()` 不带 `module.` 前缀，所以 `EMAModel.step()` 可以正确匹配参数名。

### 3. 每个 rank 使用不同 seed

```python
set_seed(cfg.training.seed + rank)
```

**原因**: 确保数据增强多样性，模型权重通过 checkpoint 加载保持一致。

### 4. 所有 rank 加载 checkpoint

```python
# 所有 rank 都加载，确保状态一致
checkpoint = workspace.load_checkpoint(resume_tag)
model.load_state_dict(checkpoint.model_state)
optimizer.load_state_dict(checkpoint.optimizer_state)
```

### 5. Normalizer 同步

```python
# rank 0 的 normalizer 广播到所有 rank
norm_state = normalizer.state_dict()
for key in norm_state:
    dist.broadcast(norm_state[key], src=0)
if rank != 0:
    normalizer.load_state_dict(norm_state)
```

### 6. Epoch 结束同步

```python
if rank == 0:
    # 验证、评估、保存 checkpoint
    pass
dist.barrier()  # 所有 rank 等待主进程完成
```

**已知问题**: 主进程执行评估时，其他 rank 的 GPU 空闲。缓解方案：减少评估频率。

## 验证机制

### 验证集处理

- **单卡**: 正常创建 val_loader
- **多卡**: 只在 rank 0 创建 val_loader
- **采样**: `shuffle=False`（确定性）

### 验证时不传入 teacher

```python
# 验证时 agent 已经是 ema_model
val_agent = self.ema_model if self.use_ema else self.model
val_loss = self.validate(val_agent)

# validate() 内部不传入 ema_model
loss_kwargs = {}  # 避免 teacher=student=ema_model
```

**原因**: 如果传入 teacher，会导致自我一致性检查，失去 distillation 意义。

## 常见问题

### Q: 为什么 EMA 模型不包装 DDP？

**A**: EMA 模型是纯推理副本，不参与梯度计算和同步。包装 DDP 会：
- 注册不必要的 all-reduce hook
- 需要通过 `.module` 访问属性，导致代码复杂
- 浪费通信带宽

### Q: 为什么 EMA 可以直接使用 DDP 模型更新？

**A**: DDP 模型的 `named_parameters()` 不带 `module.` 前缀，只有 `state_dict()` 才带前缀。`EMAModel.step()` 使用 `named_parameters()` 遍历参数，所以可以正确匹配。

### Q: 为什么每个 rank 使用不同 seed？

**A**: 确保数据增强的多样性。模型权重通过 DDP 初始化和 checkpoint 加载保持一致。

### Q: 单卡和多卡 checkpoint 兼容吗？

**A**: 完全兼容。保存时统一使用无 `module.` 前缀的 state_dict，加载时自动处理前缀。

### Q: 为什么 global_step 按 batch 计数而非优化器更新？

**A**: 用于日志频率控制。学习率调度器的 `num_training_steps` 已考虑梯度累积。

### Q: DDP 评估时其他 rank 的 GPU 为什么空闲？

**A**: 仿真环境通常不支持多进程并行。缓解方案：
1. 减少评估频率（`eval_interval_epochs: 50`）
2. 训练完成后单独运行评估脚本

## 性能优化建议

1. **减少评估频率**: `eval_interval_epochs: 50`
2. **增大 batch size**: 配合梯度累积使用
3. **使用 pin_memory**: `pin_memory: true`
4. **使用 persistent_workers**: `persistent_workers: true`
5. **调整 num_workers**: 通常设置为 CPU 核心数的 1/4
