**版本**: 1.1

## 1. 概述

仿真测评模块用于在 SAPIEN 物理仿真环境中评估灵巧手策略的成功率与效率。整体架构由三层组成：

```
eval_sim.py (入口)
    ↓
SimEvaluator (评估调度器)
    ↓
SimRunner → BaseRunner (执行引擎)
    ↓
dexmani_sim.envs.* (仿真环境)
```

**核心职责**:
- 加载训练好的 checkpoint（支持 EMA / 主模型）
- 在多个随机 seed 上 rollout 策略，录制视频
- 扫描多种 `denoise_timesteps` 配置，找到推理步数最优解
- 输出结构化指标：成功率、平均步数、逐 episode 详情、视频

## 2. dexmani_sim 环境介绍

### 2.1 基础架构

`dexmani_sim` 是基于 **SAPIEN** 物理引擎的灵巧手操作仿真环境，继承 `gymnasium.Env` 接口。

**关键文件**:

| 文件 | 职责 |
|------|------|
| `dexmani_sim/envs/base_env.py` | 基础环境类：场景构建、传感器加载、step/reset 逻辑 |
| `dexmani_sim/envs/*.py` | 具体任务实现（PickAppleMessy, Pour, MultiGrasp 等 7 个任务） |
| `dexmani_sim/sensors/` | 传感器：Camera、ContactNetForce、ImaginePointCloud |
| `dexmani_sim/robots/` | 机器人：XArm7_XHand（机械臂 + 灵巧手） |
| `dexmani_sim/objects/` | 物体：RoboTwinObject（来自 RoboTwin 数据集的 URDF/MJCF） |
| `dexmani_sim/gen_episode_data.py` | 数据采集工具：生成训练数据集和 eval seed 文件 |

### 2.2 机器人平台

**XArm7_XHand**: 7-DOF 机械臂 + 仿人灵巧手

- **动作空间**: 19 维（7 臂关节 + 12 手关节）
- **观测空间**: 多模态（RGB、深度、分割、点云、关节状态、接触力、指尖位置）
- **frame_skip**: 20（每个动作执行 20 个物理步，dt = 20 × 1/200 = 0.1s）

### 2.3 传感器与观测

```
get_sensor_data() 返回:
├── camera: dict
│   ├── rgb           (H, W, 3)       uint8
│   ├── depth         (H, W, 1)       float32
│   ├── segmentation  (H, W)          int32
│   ├── point_cloud_tensor (N, 3)    torch.Tensor
│   └── camera_config (intrinsic/extrinsic)
├── joint_state       (19,)           机器人关节位置
├── contact_force     (len(contact_links), 3)  接触力
├── fingertip_points  (len(fingertips), 3)     指尖位置
└── imagine_point_cloud (M, 3)        想象点云
```

`preprocess_obs_data()` 进一步处理：
- **实例分割**: 将原始 seg ID 映射为 {background:0, table:1, arm:2, hand:3, objects:4+}
- **点云裁剪**: crop_point_cloud 裁剪工作空间内点云，FPS 采样到 1024 点
- **展平**: camera_intrinsic (12,), camera_extrinsic (16,), contact_force (flatten), fingertip_points (flatten)

**最终 observation dict 的 key**:
`rgb`, `depth`, `segmentation`, `point_cloud`, `camera_intrinsic`, `camera_extrinsic`, `joint_state`, `contact_force`, `fingertip_points`, `imagine_point_cloud`

### 2.4 Step 流程

```python
def step(self, action):
    self.action_cnt += 1
    for _ in range(self.frame_skip):
        self.robot.balance_passive_force()
        self.robot.apply_action(action)
        self.scene_interact_with_objects()   # 子类覆写：任务特定的环境交互
        self.scene.step()
    
    # 渲染 + 获取观测
    self.scene.update_render()
    obs = self.preprocess_obs_data(self.get_sensor_data())
    
    # 录制视频帧
    self.video_frames.append(obs["rgb"])   # 或 viewer_rgb
    
    # 计算指标
    reward, done = self.compute_metric_and_condition(), self.compute_done()
    truncated = self.compute_truncated()
    return obs, reward, done, truncated, info
```

### 2.5 Done / Truncated 语义

**这是与标准 Gymnasium 最重要的区别**:

| 信号 | 触发条件 | 含义 |
|------|----------|------|
| `done` | `consistent_success_count >= 3` | 连续 3 次 step 满足成功条件 |
| `truncated` | `action_cnt >= action_cnt_limit` | 达到最大步数上限 |

**关键设计决策**: dexmani_sim **不实现 done auto-reset**。done 后继续 step 仅产生多余动作，不会 crash。这与实机 teleoperation 行为一致：无自动终止信号，走满固定步数。

因此评测循环中 `done=True` 时不 break，而是继续执行完当前 action chunk 直到 truncated。

### 2.6 Reset 机制

```python
def reset(self, seed, options):
    # 1. 重置机器人（可选随机化位姿）
    self.robot.reset(random_init=..., offset_pos=..., offset_rpy=...)
    
    # 2. 重置物体位置（子类覆写）
    self.reset_objects()
    
    # 3. 稳定性检查：800 步预热 + 200 步观察，物体旋转 > 3° 或位移 > 1cm 视为不稳定
    if not is_stable:
        raise RuntimeError(f"Reset Failed for seed {seed}! Unstable objects: ...")
    
    # 4. 初始化状态计数器
    self.initialize_env_state()  # action_cnt=0, video_frames=[], success_condition=False
    
    # 5. 返回初始观测
    return obs, info
```

**稳定性检查** (`check_stable`): reset 后运行 800 步物理预热（平衡被动力），再运行 200 步观察物体姿态变化。如果任何物体旋转变化 > 3° 或位移 > 1cm，抛出异常。这确保 episode 开始时物体已稳定。

### 3.2 BaseRunner: 执行引擎

**核心状态**:
```python
self.obs_deque = deque(maxlen=n_obs_steps)        # 观测历史
self.n_obs_steps = 2                              # 观测窗口大小
self.sensor_modalities = ["point_cloud", "joint_state"]  # 使用的传感器
```

**关键方法**:

| 方法 | 职责 |
|------|------|
| `update_obs(observation)` | 将新观测加入 deque |
| `get_stacked_obs()` | 返回最近 n_obs_steps 帧的堆叠张量，过滤 sensor_modalities |
| `get_nobs(device)` | 将 stacked_obs 转为 [1, T, ...] 格式的 torch tensor |
| `get_action_chunk(nobs, agent)` | 调用 agent.predict_action() 获取动作 chunk |
| `eval_one_episode(agent, env, seed)` | 单个 episode 的完整 rollout |
| `run(agent)` | 多 episode 评估，统计指标 |
| `make_env()` / `get_seed_list()` | 抽象方法，由子类实现 |

**观测堆叠策略** (`_stack_last_n`):

```
输入: deque 中的历史帧，n_obs_steps=2
输出: shape = (n_obs_steps, ...）的张量

情况 A: deque 已满（正常运行时）
  → 取最后 n_obs_steps 帧

情况 B: deque 不足 n_obs_steps（episode 刚开始）
  → 取所有可用帧
  → 用最早帧重复填充前面的位置（与训练数据集 pad 一致）

支持类型:
  - np.ndarray: np.zeros + np.asarray 填充
  - torch.Tensor: torch.zeros + torch.stack 填充
  - str: [last_value] * n_steps
```

**训练-评估一致性**: Dataset 中 `SequenceSampler` 对 horizon 窗口不足时使用最早帧 padding（sampler.py:165），评估时的 `_stack_last_n` 使用相同的 padding 策略，确保训练和评估的输入分布一致。

### 3.3 SimRunner: 环境适配层

```python
class SimRunner(BaseRunner):
    task_name: str          # "pick_apple_messy"
    env_kwargs: dict        # 传递给环境构造的额外参数
    eval_seeds: List[int]   # 可选，直接指定 seed 列表（跳过文件读取）
    
    make_env():
        module = importlib.import_module(f"dexmani_sim.envs.{task_name}")
        class_name = name_to_pascal_case(task_name)   # "pick_apple_messy" → "PickAppleMessy"
        return module.PickAppleMessy(render_mode="rgb_array", **env_kwargs)
    
    get_seed_list():
        if self.eval_seeds is not None:
            return self.eval_seeds          # 直接使用传入的 seed 列表
        seed_file = DATA_DIR / "eval_seeds" / f"{task_name}.txt"
        if seed_file.exists():
            return [int(x) for x in seed_file.read_text().split()]
        return list(range(100))             # fallback: seed 0-99
```

**命名转换**: 蛇形命名 → 帕斯卡命名。`pick_apple_messy` → `PickAppleMessy`，`stack_cups` → `StackCups`。

**Seed 来源优先级**: `eval_seeds` 参数 > `eval_seeds/{task_name}.txt` 文件 > `range(100)` fallback。

### 3.4 MultiTaskSimRunner: 多任务评估

**文件**: `env_runner/multi_task_sim_runner.py`

为多任务策略提供逐任务评估能力。

**TaskTextSimRunner** (继承 `SimRunner`):
- override `get_action_chunk()`，在调用 `agent.predict_action()` 前将 `task_text` 注入 `obs_batch`
- 使 MultiTaskAgent 能根据 task_text 生成对应任务的动作

```python
class TaskTextSimRunner(SimRunner):
    def get_action_chunk(self, obs_batch, agent, denoise_timesteps=None):
        obs_batch["task_text"] = [self.task_text]  # 注入任务文本
        return super().get_action_chunk(obs_batch, agent, denoise_timesteps)
```

**MultiTaskSimRunner**:
- 为每个 task 创建独立的 `TaskTextSimRunner` 实例
- 逐 task 运行 `runner.run()`，汇总各 task 的 `success_rate` 取平均
- 异常处理：单个 task 失败不影响其他 task 的评估

```python
class MultiTaskSimRunner:
    def run(self, agent, denoise_timesteps=None, eval_episodes=None):
        per_task = {}
        for task_name, runner in self.runners.items():
            result = runner.run(agent, denoise_timesteps, eval_episodes)
            per_task[task_name] = result
        
        rates = [r["success_rate"] for r in per_task.values() if r["success_rate"] is not None]
        avg_success_rate = sum(rates) / len(rates) if rates else None
        return {"success_rate": avg_success_rate, "per_task": per_task, ...}
```

## 4. 评测流程

### 4.1 完整流程图

```
1. 解析参数: --policy-name, --task-name, --exp-name, overrides
   ↓
2. 构建 exp_dir: experiments/{policy}/{task}/{exp}
   ↓
3. 加载配置: config.yaml + overrides
   ↓
4. 设置随机种子: set_seed(cfg.eval.seed)
   ↓
5. 实例化组件: agent, env_runner, workspace, device
   ↓
6. 创建 SimEvaluator
   ↓
7. 加载 checkpoint: workspace.load_for_inference(model=agent, tag=ckpt_tag, use_ema=True)
   ↓
8. 创建评估运行目录: eval/{YYYYMMDD_HHMMSS}/
   ↓
9. for denoise_timesteps in [5, 10, 20, ...]:
   │   │
   │   ├── env_runner.run(agent, denoise_timesteps, eval_episodes)
   │   │   │
   │   │   ├── make_env()                    # 创建仿真环境
   │   │   ├── get_seed_list()               # 读取 seed 文件
   │   │   │
   │   │   └── for episode in seeds[:eval_episodes]:
   │   │       │
   │   │       ├── env.reset(seed)
   │   │       ├── runner.reset()            # 清空 deque
   │   │       ├── runner.update_obs(obs)    # 加入初始观测
   │   │       │
   │   │       └── while not truncated:
   │   │           ├── nobs = runner.get_nobs()          # [1, T, ...]
   │   │           ├── action_chunk = agent.predict_action()  # [chunk_len, action_dim]
   │   │           └── for action in action_chunk:
   │   │               ├── obs, reward, done, truncated, info = env.step(action)
   │   │               ├── runner.update_obs(obs)
   │   │               └── if done: record task_done_step
   │   │
   │   │   ├── 统计: success_rate, avg_steps, episode_details
   │   │   └── env.close()
   │   │
   │   ├── 保存视频: eval_episode_{idx}_seed_{seed}.mp4
   │   └── 保存指标: denoise_timesteps{N}/metrics.json
   │
10. 保存汇总: eval_metrics.json
    ↓
11. 完成
```

### 4.2 单 Episode 详细时序

```
time ──────────────────────────────────────────────────────────►

reset(seed)
    │
    ├─ obs_0, info_0
    │
update_obs(obs_0)                     # deque: [obs_0]
    │
    │  step 1:
    │  get_nobs()                      # deque: [obs_0] → stack pad → [1, 2, ...]
    │  predict_action()                # → action_chunk [8, 19]
    │  ┌─ step(action_chunk[0])        # → obs_1, reward, done, truncated
    │  │  update_obs(obs_1)            # deque: [obs_0, obs_1]
    │  ├─ step(action_chunk[1])        # → obs_2
    │  │  update_obs(obs_2)            # deque: [obs_0, obs_1, obs_2]
    │  ├─ ...
    │  └─ step(action_chunk[7])        # → obs_8
    │     update_obs(obs_8)
    │
    │  step 2:
    │  get_nobs()                      # deque: [..., obs_8] → stack last 2 → [1, 2, ...]
    │  predict_action()                # → action_chunk [8, 19]
    │  ┌─ step(action_chunk[0])
    │  │  ...
    │  └─ step(action_chunk[7])
    │
    │  step N:
    │  truncated = (action_cnt >= action_cnt_limit)  → break
    │
    └─ return done, task_done_step
```

**关键**: 每次 `predict_action` 输出一个 action chunk（默认 8 步），逐个执行并更新观测。chunk 内的动作是开环执行的，chunk 之间通过新的观测重新预测，形成闭环。

## 5. 指标定义

### 5.1 核心指标

| 指标 | 计算方式 | 含义 |
|------|----------|------|
| `success_rate` | `mean(success_list)` | 任务完成率（done=True 的比例） |
| `avg_steps` | `mean(task_done_step_list)` | 成功 episode 的平均完成步数 |

**None 语义**: 当没有有效 episode 时（seed 全部失败），`success_rate` 和 `avg_steps` 都返回 `None`，而非 `0.0`。这区分了"全失败"和"无数据"两种情况。

### 5.2 task_done_step 记录逻辑

```python
if done and not prev_done:
    task_done_step = env.action_cnt   # 首次检测到 done 时的步数
prev_done = done
```

只在 done 从 False 变为 True 的瞬间记录步数，避免重复记录。如果 done 始终为 False，`task_done_step` 保持 `None`。

### 5.3 异常分类统计

| 异常类型 | 处理方式 | 计入 success_list |
|----------|----------|-------------------|
| env init 失败（Reset Failed / Unstable / Failed to sample） | 跳过 seed，不计入 | 否 |
| policy 推理异常（RuntimeError / ValueError，非 env init） | 记为失败 | 是 (False) |
| 用户中断（KeyboardInterrupt） | 优雅退出，汇总已收集结果 | 中断前已收集的保留 |
| 未预期异常（其他 Exception） | 打印 traceback 后重新抛出 | 不计入（程序终止） |

**env init 失败检测**基于 dexmani_sim 的错误消息字符串匹配：
- `base_env.py:316` — `"Reset Failed for seed {seed}! Unstable objects: ..."`
- `base_env.py:429/473` — `"Failed to sample ... positions ..."`

## 6. 训练时评测

训练过程中，`Trainer` 在每个 epoch 结束时根据 `eval_interval_epochs` 触发评测：

```python
# trainer.py: plan_epoch_end_tasks()
should_evaluate = is_last_epoch or (epoch_idx % self.eval_interval_epochs == 0)

# trainer.py: evaluate()
result = self.env_runner.run(agent)   # 只跑一次 denoise_timesteps（默认值）
metrics = {
    "eval/success_rate": success_rate * 100 if success_rate is not None else None,
    "eval/avg_steps": result["avg_steps"],
}
```

**与独立评估的区别**:
- 训练时只用单一 `denoise_timesteps`（agent 默认值）
- 独立评估扫描多种 `denoise_timesteps` 配置
- 训练时评测结果用于 checkpoint 的 `test_mean_score` 排序

## 8. 关键设计决策

### 8.1 为什么 done 后不 break？

dexmani_sim 的 `compute_done()` 只检查 `consistent_success_count >= 3`，不触发 auto-reset。done 后继续 step 只是产生多余动作，不会 crash。这与实机 teleoperation 行为一致。

### 8.2 为什么 obs_deque 大小为 n_obs_steps？

`deque(maxlen=n_obs_steps)` 恰好保存最近 `n_obs_steps` 帧观测，与训练时的 `n_obs_steps` 窗口一致。`BaseRunner.stack_last_n()` 在 deque 不足 `n_obs_steps` 时（episode 刚开始），会用最早帧重复填充，与训练数据集的 padding 策略一致（`sampler.py:165`）。

### 8.3 为什么评估时使用 EMA 模型？

EMA（指数移动平均）模型在训练过程中维护，通常比主模型更稳定、泛化更好。`use_ema_for_eval=True` 是默认配置。

### 8.4 为什么扫描多种 denoise_timesteps？

对于扩散模型和流匹配模型，推理步数影响速度和质量。扫描多个值可以找到 Pareto 前沿：用多少步能获得可接受的成功率。

## 9. 与训练数据集的关系

| 维度 | 训练数据 | 评估 |
|------|----------|------|
| 数据来源 | MimicGen 生成轨迹 | dexmani_sim 策略 rollout |
| 随机性 | seed 决定物体位姿 | seed 决定物体位姿 |
| 观测窗口 | SequenceSampler 按 horizon 采样 | deque 维护滑动窗口 |
| Padding | 最早帧重复 | 最早帧重复（一致） |
| 归一化 | LinearNormalizer limits | 使用训练时 fitted 的 normalizer |

**一致性保证**: 评估时复用训练时 fitted 的 `LinearNormalizer`，确保观测和动作的归一化范围一致。点云和 RGB 不归一化。
